1. Executive Summary & Core Philosophy
Project Atlas is the foundational data engine for Dossily—a strictly factual, citation-backed digital library of visa requirements tailored exclusively for Indian passport holders.

The Mission: Function as the "Wikipedia of Visas." Solve the Indian travel market's fragmentation and SEO pollution by providing a zero-fluff, single source of truth.

Core Philosophy: "Trust, but Verify." Every metric, fee, and requirement must map to an official government circular or embassy portal.

Strategic Goal: Capture high-intent organic traffic, establish absolute domain authority, and build the meticulously structured database required for Vertical 2 (The Automated AI Analysis Tool).

2. Core Technology Stack
This project strictly avoids standard Single Page Application (SPA) or database-driven architectures to minimize UI maintenance and hosting complexity.

Static Site Generator: MkDocs (Python-based).

UI/Theme: Material for MkDocs (Requires zero UI coding; strictly managed via mkdocs.yml).

Generation Engine: mkdocs-gen-files (Python plugin to dynamically generate virtual Markdown files).

Templating: Jinja2 (Maps data to Markdown).

Hosting/CI-CD: GitHub Actions to GitHub Pages (or Cloudflare/Netlify) with automated link-rot testing.

3. The Data Architecture (Strict Separation of Concerns)
Atlas operates on a strict "Content is Downstream of Data" philosophy. We do not write Markdown files for country pages.

The Source of Truth (data/visas/*.yaml): All factual data lives in structured YAML files. This acts as the database.

The Template (templates/country.md.jinja): The layout schema that dictates how sections (Authority, Indian Privilege, Hard Facts) are displayed.

The Engine (gen_pages.py): A Python script that runs at build time. It intercepts the MkDocs process, loops through the YAML data, injects it into the Jinja2 template, and creates virtual Markdown pages in memory for MkDocs to render.

AI Directive: Never suggest modifying .md files in the docs/ folder for visa pages. All structural changes must occur in the Jinja template, and all factual changes must occur in the YAML data files.

4. The Data Schema Contract
Every country YAML file must adhere to this exact structure to ensure machine-readability for Vertical 2:

country: Name of the destination.

iso_code: 2-letter country code.

authority: Object containing name, processor (e.g., VFS), and strictly verified official_portal URL.

requirements: Object containing visa_fee_inr, processing_days, photo_specs (dimensions, bg_color), and financial_proof.

(Future expansions will include transit_policy, jurisdiction_maps, and third_party_exemptions).

5. SEO & Semantic Metadata
To capture "Zero-Click" searches and Google Knowledge Panels, the site injects Schema.org JSON-LD directly into the HTML <head> via the Jinja2 template.

GovernmentService: Maps the "Visa Application Service" to the embassy/consulate.

GovernmentPermit: Maps the visa document itself, its validity, and the issuing authority.

6. Community Workflow & CI/CD Pipeline
To prevent founder burnout, the platform relies on community audits ("Edit on GitHub" mechanism).

Edit Routing: The gen_pages.py script reroutes the MkDocs "Edit this page" button directly to the underlying country.yaml file via mkdocs_gen_files.set_edit_path(). Contributors edit raw data, not UI code.

Pull Requests: A user spots a fee change, edits the YAML file on GitHub, and submits a PR with a citation link.

Continuous Integration: GitHub Actions automatically tests the build.

Link Health (Zero Dead Links): An htmlproofer step runs during CI/CD to crawl all official embassy links. If a government URL rots, the build fails and alerts the Librarian.

7. Extended Ecosystem (P2 Roadmap)
The Changelog / Blog: Utilizing the native Material for MkDocs blog plugin to maintain an immutable history of rule changes (e.g., "Schengen Fee Increased - Feb 2026").

The Team-BHP Style Forum: Integrating Giscus. This embeds GitHub Discussions directly into the static site, providing a robust community forum without requiring a backend database or user authentication system.

Interactive UI/UX: A JavaScript-driven SVG world map on the homepage. The Python engine will export a status.json file during the build, allowing the frontend JS to dynamically color-code countries (e.g., Visa-free vs. eVisa) based on the current YAML data.
